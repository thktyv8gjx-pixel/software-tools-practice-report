摘要

本实验以“从本地开发到Linux服务器部署”为目标场景，在Windows 11中搭建Node.js、npm、Git、Visual Studio Code和Hexo开发环境，使用Markdown完成个人实验报告静态站点；通过Git记录项目版本并同步至GitHub；在VirtualBox中安装Ubuntu Server 22.04.5 LTS，配置NAT与Host-Only双网卡、Linux用户与权限、Shell环境变量、OpenSSH远程管理和裸Git仓库；最终使用Nginx提供静态Web服务，并编写PowerShell部署脚本实现Hexo构建、SCP上传、rsync同步、HTTP状态验证与双远程仓库推送。报告重点讨论可复现环境、最小权限、网络职责分离、密钥认证、版本快照和自动部署等工程配置，同时基于官方技术文档提出由“虚拟机+手工同步”升级为“容器化+GitHub Actions持续部署”的可行方案。

1 实验目的

1.1 总体目标

本课程实验旨在综合掌握现代软件开发工具的安装、配置和协同使用方法，独立完成从内容编写、代码版本管理、虚拟化Linux环境构建、账户权限配置、远程运维到Web静态站点部署的全过程。通过连续性的综合任务，形成可复现、可追踪、可维护和可部署的软件工程实践能力。[1-2]

1.2 目标场景

本人的目标场景为：在Windows 11宿主机上使用VS Code编写Markdown实验报告，由Node.js、npm和Hexo生成静态HTML；使用Git在本地形成版本快照，并同时推送至GitHub与Ubuntu裸Git仓库；使用VirtualBox运行Ubuntu Server，通过NAT网卡连接互联网、通过Host-Only固定地址192.168.56.101接受SSH与HTTP访问；使用SCP和rsync同步文件，由Nginx监听80端口提供静态站点；最后通过PowerShell脚本串联清理、构建、上传、同步和验证步骤。

1.3 分项目标

  1. 完成Web开发环境安装并运行Hexo静态站点，理解运行时、包管理器、依赖清单和静态站点生成器的关系。

  2. 使用VS Code和Markdown编写结构化文档，掌握Git工作区、暂存区、本地仓库、远程仓库及提交历史。

  3. 使用VirtualBox安装Ubuntu Server，配置虚拟硬件、NAT与Host-Only双网卡以及固定IP。

  4. 掌握Linux文件导航、内容查看、软件包管理、资源查看等基本命令。

  5. 完成用户、用户组、Home目录、文件所有权和权限控制，理解最小权限原则。

  6. 配置用户级和系统级环境变量、别名、PATH及自定义命令，并验证重新登录后的持久性。

  7. 实现SSH远程登录、SCP传输、ED25519密钥认证、快捷主机配置和Ubuntu裸Git仓库。

  8. 安装配置Nginx，将Hexo静态文件部署至服务器，并通过自动化脚本和浏览器验证部署结果。

图16 Ubuntu Server虚拟机硬件配置

图17 VirtualBox的NAT与Host-Only双网卡设置

安装完成后系统识别enp0s3和enp0s8。编辑/etc/netplan/50-cloud-init.yaml，使NAT网卡继续使用DHCP，Host-Only网卡固定为192.168.56.101/24。Netplan文件使用YAML格式，必须严格采用空格缩进，且Host-Only接口不配置默认网关。[8-9]

3.5 实验五：Linux账户与权限管理

3.5.1 创建用户与Home目录

使用sudo adduser webadmin创建部署用户。系统自动生成UID、GID、/home/webadmin和/bin/bash登录Shell。通过id、getent passwd和ls -ld核对账户、主组和Home目录。

图30 webadmin用户、UID/GID与Home目录创建结果

3.5.2 创建项目组和共享目录

创建webgroup，将renyaocheng和webadmin追加到该组，并建立/srv/software-tools-site作为共享网站目录。目录属主为root、属组为webgroup，权限设置为2770。数字2表示setgid，新建文件将继承webgroup；770表示所有者和组可读写进入，其他用户无权限。

图31 webgroup成员及共享目录2770权限

3.5.3 多用户读写验证

renyaocheng先在共享目录创建ren-file.txt，再通过su - webadmin切换账户，由webadmin读取前一文件并创建新文件。两个用户均能访问共享目录，且文件继承webgroup，实现了组级协作而不开放给无关用户。

图32 webadmin访问共享目录并创建文件

3.5.4 脚本执行权限

创建deploy-test.sh后首次直接执行出现Permission denied，原因是文件仅有读写权限。执行chmod u+x增加所有者执行位后，权限由-rw-rw-r–变为-rwxrw-r–，脚本正常输出Deploy test successful。

图33 脚本增加执行权限前后的对比

3.5.5 本实验结果

5.2 重点配置及工程意义

重点配置 关键内容 工程意义
npm镜像 registry=https://registry.npmmirror.com 改善依赖下载可用性，但应保留切回官方源的能力
.gitignore 排除node_modules、public、日志 减少仓库体积，避免提交可生成文件和临时数据

5.3 关键原则

  1. 可复现：通过版本输出、依赖锁文件、配置文件和脚本记录环境，而不是依赖口头说明。

  2. 可追踪：每次提交聚焦一个目标，提交说明能够解释变更原因。

  3. 最小权限：不使用chmod -R 777解决问题，而是设计用户组、属主和必要权限。

  4. 职责分离：NAT负责外网，Host-Only负责实验内网；源码目录和生成目录分离；用户级与系统级配置分离。

  5. 先检查再生效：YAML、SSH和Nginx配置在重载服务前分别使用netplan generate、sshd -t和nginx -t检查。

  6. 自动化必须可验证：脚本不仅执行部署，还使用curl检查HTTP返回码，并保留Git提交记录。

6 出现问题及解决方案

实验过程中的错误并非额外负担，而是理解工具加载机制、环境变量、网络启动和配置格式的重要材料。以下问题均来自本人实际操作，并保留典型错误截图。

6.1 Node.js命令无法识别

执行node -v时PowerShell提示CommandNotFoundException。原因是Node.js尚未安装或安装目录未加入PATH。安装Node.js并保留Add to PATH，关闭旧终端后重新打开，node -v正常输出v24.18.0。

图56 典型问题 1 Node.js命令无法识别

6.2 PowerShell阻止npm.ps1

node可用但npm -v提示系统禁止运行脚本。原因是PowerShell执行策略阻止npm.ps1。执行Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned并重新打开终端后恢复；npm.cmd -v可用于确认npm本体是否安装。

图57 典型问题 2 PowerShell阻止npm.ps1

第一次将地址写成localhost4000，且Hexo服务已退出，浏览器显示ERR_CONNECTION_REFUSED。改为http://localhost:4000，并保持hexo server进程运行后解决。

图58 典型问题 3 localhost页面拒绝连接

VirtualBox启动时提示Could not read from the boot medium。原因是Ubuntu ISO未挂载。关闭虚拟机，在“设置-存储”中将ISO挂载到虚拟光驱并设置光驱优先启动。

图59 典型问题 4 虚拟机无法读取启动介质

首次安装在curtin in-target/grub-pc阶段失败。删除失败虚拟机，重新创建动态VDI，关闭EFI与3D加速，使用完整Ubuntu Server 22.04.5 LTS镜像并避免安装阶段在线升级，第二次安装成功。

图60 典型问题 5 Ubuntu安装在GRUB阶段失败

netplan generate提示Invalid YAML: aliases are not supported，原因是误输入字符和缩进不规范。清空文件后使用空格重新输入，执行chmod 600、netplan generate和netplan apply，固定IP生效。

图61 典型问题 6 Netplan YAML解析失败

图63 典型问题 8 Hexo误加载PowerShell部署脚本

6.9 VirtualBox克隆时磁盘空间不足

现象：完整克隆node2、node3时出现VERR_DISK_FULL，部分节点条目已经创建但虚拟磁盘不完整，无法正常启动。原因是VirtualBox默认虚拟机目录所在磁盘剩余空间不足，而完整克隆会复制整套虚拟磁盘。处理时先删除失败节点并选择“删除所有文件”，清理无效VDI和临时文件；随后释放宿主机磁盘空间，再逐台克隆并在每次完成后检查虚拟磁盘路径与容量。该问题说明虚拟化实验前应预估模板机数量与磁盘开销，不能只关注虚拟机的动态扩展上限。

图64 典型问题9：完整克隆因宿主机磁盘空间不足失败

6.10 克隆节点启动卡住与SSH超时

现象：克隆后的node1一度停留在“Loading essential drivers”附近，Windows侧对192.168.56.102的ping和22端口测试均超时。排查顺序为：确认虚拟机不是处于保存状态；检查两个网卡均已启用且网线已连接；进入GRUB正常启动项；登录后核对hostname、ip -br addr和systemctl is-active ssh。最终发现节点仍继承模板身份与旧网络状态，重新生成machine-id及SSH主机密钥、确认Host-Only地址并重启后恢复。该过程强调应先分层判断“虚拟机是否启动—网卡是否有地址—SSH服务是否运行”，避免把所有超时都归因于防火墙。

图65 典型问题10：克隆节点启动停滞并导致远程连接超时

6.11 GitHub Pages主题CSS返回404

现象：Pages首页能够打开，但只显示浏览器默认排版；直接访问/css/style.css返回404。排查发现Hexo本地已生成public/css/style.css，gh-pages分支也包含该文件，因此问题不在Hexo和Actions构建层。最终定位为GitHub实际仓库名曾拼写为“sofware-tools-practice-report”（software少一个t），而_config.github.yml和HTML资源路径使用正确的“software-tools-practice-report”，仓库路由与资源根路径不一致。将仓库重命名为正确拼写、更新origin并重新触发部署后，CSS请求恢复200，公网完整主题正常显示。该故障说明部署排查必须同时核对仓库名、Pages路径、Hexo url/root和HTML资源引用，不能只依据工作流绿色状态判断成功。

本次实践最大的收获，不是简单记住若干命令，而是第一次真正把编辑器、构建工具、版本控制、虚拟机、Linux、网络、远程管理和Web服务器串联成一条完整的工程链路。过去我会把“写Markdown”“提交Git”“登录服务器”理解为互不相关的操作；经过本次实验后，我能够清楚说明每个工具的输入与输出：Markdown是内容源，Hexo将其构建为静态资源，Git保存可追踪版本，SSH承担安全传输，Linux权限保护部署目录，Nginx最终向浏览器提供服务。

这种从局部操作走向系统理解的变化，对我帮助很大。面对一个新项目时，我不再只关注“代码能不能运行”，而会主动确认运行环境、依赖版本、目录结构、网络入口、权限边界和部署验证方式。尤其是亲手完成本地localhost预览、GitHub远程备份、Ubuntu服务器发布和自动部署脚本后，我对“软件交付”有了更具体的认识：一个成果只有能够被别人复现、部署和验证，才真正具备工程价值。

排错过程也让我形成了更明确的方法：先观察错误原文和命令返回码，再判断问题属于路径、权限、服务、网络还是配置格式；修改后使用专用检查命令验证，而不是反复尝试。例如Netplan、SSH和Nginx分别有generate、sshd -t和nginx -t，部署完成还需要curl和浏览器双重确认。

7.2 工程思维与故障定位

实验中出现的多次故障，让我逐渐形成了较为清晰的排错思路：先完整阅读错误信息和命令返回结果，再判断问题属于路径、权限、服务状态、网络连通性还是配置格式；修改之后使用专用检查命令验证，而不是继续盲目尝试。例如，Netplan、SSH和Nginx分别可以通过netplan generate、sshd -t和nginx -t进行语法检查，部署后还要通过curl与浏览器进行双重确认。

我也认识到，很多错误并不是工具本身复杂，而是操作时忽略了细节和约定。例如git参数前漏写空格、localhost地址漏写冒号、YAML缩进不规范、PowerShell脚本放入Hexo具有特殊含义的scripts目录，都会导致看似严重的问题。今后遇到报错时，我会优先保留现场、记录命令、缩小问题范围，再做最小修改并复测；同时在修改系统配置前先备份文件，使排错过程可回退、可复盘。

本次补充实验让我更明确地认识到，故障定位不能依赖“反复重试”，而应按层次建立证据：虚拟机问题先确认磁盘和启动状态，网络问题依次检查接口、地址、连通性和端口，SSH问题再检查服务与认证，Web问题则分别核对构建产物、发布分支、Pages来源和资源URL。特别是GitHub Actions多次显示绿色但CSS仍404时，最终通过直接访问Raw文件、检查仓库真实名称和资源根路径才定位到拼写不一致。这一过程显著提升了我使用日志、状态命令和最小验证用例缩小问题范围的能力。

7.3 安全责任与规范意识

账户与权限实验使我认识到，“能运行”并不是软件工程的唯一标准。直接开放777权限、将私钥或Token上传到仓库、长期使用root账户登录，虽然可能暂时省事，却会把局部问题转化为更大的安全风险。本实验采用webgroup共享组、2770目录权限、SSH公钥认证以及独立配置文件，体现了最小权限、敏感信息隔离和职责分离的基本原则。

自动化同样伴随着责任。部署脚本中的rsync –delete可以保持目标目录整洁，但如果源路径或目标路径写错，也可能快速删除重要文件；因此自动化流程必须配置明确的目录、错误中止机制和结果检查。作为软件工程专业学生，我需要对自己编写和执行的命令负责，不仅关注功能结果，还应重视数据安全、服务稳定、配置审计和故障回滚。

7.4 开源文化与国产软件生态

Hexo、Git、VS Code、VirtualBox、Ubuntu、OpenSSH和Nginx等工具，使个人能够以较低成本完成从内容开发到服务器发布的完整流程。我在使用这些工具的过程中切实体会到，开源并不只是“免费使用”，更重要的是公开文档、开放标准、社区协作和持续改进。许多问题都可以通过阅读官方文档、查看日志和参考社区经验解决，这种知识共享显著降低了学习和创新门槛。

与此同时，国内已经形成Gitee、Coding、openEuler、龙蜥等平台和操作系统生态。国产软件生态的发展既需要核心技术突破，也需要高质量文档、稳定工具链、开放社区和真实用户反馈。今后的学习中，我愿意从规范提交Issue、完善中文文档、参与开源项目测试等力所能及的工作做起，不只做工具的使用者，也逐步成为可靠的贡献者。

7.5 个人反思与后续学习计划

回顾整个实践过程，我从最初连node、npm和Git命令都需要逐项确认，逐步完成了虚拟机安装、Linux网络配置、用户权限管理、SSH免密登录、自建Git仓库和Nginx自动部署。这种进步让我意识到，技术能力往往不是一次性掌握的，而是在持续操作、犯错、验证和总结中形成。能够独立把一个静态站点部署到自己配置的Linux服务器，并通过脚本重复完成发布，是本次课程带给我最直观的成就感。

后续我计划在当前成果上继续学习Docker和GitHub Actions，将本次手动配置过的构建环境、服务配置和部署流程进一步容器化、自动化；同时补充Shell脚本、Linux网络、安全加固和CI/CD知识。更重要的是，我会延续本次实践形成的习惯：为项目维护README和版本记录，重要操作前备份，提交前检查敏感信息，部署后进行健康验证。只有把规范落实到每一次小操作中，才能逐步形成稳定、可信的软件工程能力。

从单台Ubuntu扩展到三个节点后，我体会到“可复制环境”并不等于“复制完成即可使用”。每个克隆节点还必须具备独立主机名、machine-id、SSH主机密钥和静态地址；跨节点管理也必须同时考虑认证、文件一致性和删除同步。GitHub Actions与Pages的实际落地，则让我第一次完整经历了由一次Git提交自动触发构建与发布的CI/CD闭环。后续我计划继续学习Docker、反向代理、HTTPS证书、Secrets管理和自动回滚，将本次静态站点实践逐步扩展为可测试、可审计、可恢复的工程化部署流程。

8 技术展望：容器化与持续部署升级方案

8.1 代表性工具与替代选项

本报告选择“VirtualBox中的Ubuntu + 主机安装Nginx + PowerShell/SCP/rsync手工触发部署”作为代表性实践链路。该方案非常适合学习Linux、网络和服务器配置，但在多项目、多人协作和长期运行场景中，完整虚拟机占用资源较多，环境复制速度较慢，应用配置与操作系统耦合，部署仍依赖个人电脑触发。

现有工具/环节 替代或升级选项 适用特点
VirtualBox完整虚拟机 Docker、Podman、WSL2、Hyper-V 容器更轻量；WSL2适合本地Linux开发；Hyper-V适合Windows原生虚拟化
Hexo Hugo、VitePress、Astro、Jekyll Hugo构建快；VitePress适合文档；Astro适合现代前端；Jekyll与GitHub Pages集成成熟
Nginx Caddy、Apache HTTP Server、GitHub Pages Caddy自动HTTPS；Apache模块丰富；Pages免维护服务器
PowerShell手动触发 GitHub Actions、GitLab CI、Jenkins 代码推送后自动构建、测试和部署
SCP全量上传 rsync、制品仓库、容器镜像仓库 增量传输或将可部署制品版本化

图67 由虚拟机手工部署升级为容器化持续部署

8.2 升级目标

  1. 将Node.js、Hexo和Nginx运行环境写入Dockerfile或Compose文件，使构建和运行环境可复制。

  2. 以Git push作为唯一触发入口，由GitHub Actions自动安装依赖、生成站点、检查输出并构建镜像或发布制品。

  3. 使用GitHub Environment和Secrets保存服务器私钥、主机地址等敏感信息，不在仓库中明文保存。

  4. 部署前后执行健康检查，失败时自动停止发布或回滚到上一镜像/制品。

  5. 保留Podman作为无守护进程、可Rootless运行的替代容器引擎，降低长期守护进程和root权限依赖。

8.3 详细实施方案

8.3.1 第一阶段：构建环境容器化

采用多阶段Dockerfile。第一阶段使用Node.js LTS镜像安装npm依赖并执行npx hexo generate；第二阶段使用官方Nginx镜像，仅复制public目录和Nginx配置。构建镜像不包含源码、node_modules和私钥，最终运行层体积更小、攻击面更低。[12]

8.3.2 第二阶段：GitHub Actions持续集成

工作流在main分支推送时触发，依次检出代码、安装Node.js、执行npm ci、运行Hexo构建、检查public/index.html、构建容器镜像。Actions能够将构建过程固定在干净运行器中，避免“只在本人电脑上能运行”。[13]

8.3.3 第三阶段：安全部署与回滚

部署Job使用environment: production，敏感信息放入Secrets；通过SSH在服务器拉取指定提交对应的镜像并启动新容器。新容器先映射到临时端口进行健康检查，确认200后再切换Nginx反向代理或替换旧容器。保留上一版本镜像标签last-good，若健康检查失败则恢复旧容器。

  • 分支保护：main需通过构建检查后才能合并。

  • 并发控制：production同一时间只允许一个部署任务。

  • 审批机制：关键环境可配置人工批准。

  • 密钥管理：私钥、Token、服务器地址放入Secrets，日志中不输出敏感值。

  • 回滚策略：镜像以Git SHA标记，保留last-good和最近若干版本。

  • 健康检查：部署前test -f public/index.html，部署后curl –fail访问站点。

8.4 可行性、风险与收益

方面 预期收益 主要风险与控制
环境一致性 开发、CI和服务器使用同一镜像定义 固定基础镜像版本；定期更新依赖并扫描漏洞
部署效率 推送后自动构建和发布，减少手工步骤 设置审批、并发限制和健康检查，防止错误自动扩散
回滚能力 镜像和Git SHA一一对应，可快速恢复 保留last-good标签和部署日志
资源占用 容器比完整虚拟机启动更快、占用更低 教学阶段仍保留VirtualBox以学习Linux与网络基础
安全性 Secrets集中管理，容器可Rootless运行 最小权限、只读文件系统、限制网络与端口

8.5 网络调研方法

技术展望采用多源交叉验证：第一，阅读Docker、Podman、GitHub Actions、GitHub Pages和Nginx官方文档，确认工具定位、工作流语法与部署能力；第二，对比现有实验中的真实痛点，包括虚拟机资源占用、手工步骤、环境差异与回滚困难；第三，以最小可行原型设计Dockerfile和Actions流程，并用构建产物存在性、HTTP 200、Git SHA标签等可验证指标评估方案；第四，保留替代工具和风险分析，避免把单一工具描述为唯一答案。[12-13]

8.6 展望结论

VirtualBox+Ubuntu+Nginx是理解操作系统、网络和权限的必要学习路径，不应被容器完全替代；容器化和GitHub Actions则适合把已掌握的手工流程标准化、自动化。合理升级方式是“先用虚拟机理解底层，再用容器固化应用环境，最后用CI/CD控制交付”，从而兼顾基础能力与工程效率。
