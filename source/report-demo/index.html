Project Overview

项目概览

本项目以“从本地开发到 Linux 服务器部署”为目标场景，把 Windows 开发、Hexo 静态构建、Git 版本管理、Ubuntu Server 运维、Nginx 发布和 GitHub Pages 公网部署串联为一条完整证据链。

0项实验

按 Word 原顺序组织，从 Web 环境到 Web 软件部署。

0项考核核心

项目配置、Web 运行环境、Linux、账户、远程管理与部署。

0Ubuntu 节点

ubuntu-server、node1、node2、node3 组成多节点实验环境。

Dual Channel Architecture

双通道技术架构

私网通道负责课程实验环境内的 Ubuntu Server 与 Nginx 发布，公网通道使用 GitHub main 分支、Actions、gh-pages 分支与 Pages 完成互联网访问。

私网通道

  1. Windows 11 开发主机
  2. VS Code / Markdown / Hexo
  3. Hexo 静态文件
  4. SSH / SCP / rsync
  5. Ubuntu Server
  6. Nginx
  7. 192.168.56.101

Ubuntu Server 固定 Host-Only 地址，Nginx 监听 80 端口，部署脚本以 HTTP 200 作为验收信号。

公网通道

  1. Windows 11 开发主机
  2. Git 本地仓库
  3. GitHub main 分支
  4. GitHub Actions
  5. gh-pages 分支
  6. GitHub Pages
  7. 公网访问

main 保存源码，gh-pages 保存生成后的静态网站，Pages 使用 gh-pages 根目录作为发布来源。

6.1 Node.js命令无法识别

问题现象

Node.js 尚未安装或安装目录未加入 PATH。
排查过程
确认命令不可用、重新安装并保留 Add to PATH，关闭旧终端后重开。
解决操作
安装 Node.js 后重新验证 node -v。
成功截图
图3
最终结果
node -v 正常输出 v24.18.0。

6.2 PowerShell阻止npm.ps1

问题现象
node 可用但 npm -v 被执行策略阻止。
错误截图
图57
原因分析
PowerShell 执行策略阻止 npm.ps1。
排查过程
用 npm.cmd -v 判断 npm 本体是否存在，再调整当前用户执行策略。
解决操作
执行 Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned。
成功截图
图3
最终结果
npm 命令恢复，后续可安装 Hexo CLI。

解决操作

成功截图

本地页面可正常显示。

实验总结

Word 第 7 章强调，本次实践的价值不只是记住命令，而是把编辑器、构建工具、版本控制、虚拟机、Linux、网络、远程管理和 Web 服务器串联成完整工程链路。

Markdown 是内容源，Hexo 生成静态资源，Git 保存可追踪版本，SSH 承担安全传输，Linux 权限保护部署目录，Nginx 向浏览器提供服务。

故障定位方法

安全与规范意识
